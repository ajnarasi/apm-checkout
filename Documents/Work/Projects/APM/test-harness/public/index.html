The Problem We Solved

Every payment method used to be a standalone 4-6 week engineering project: read the provider docs, build the integration, handle errors, write tests, deploy. With 55 APMs required for global coverage, that's 212-318 engineering weeks of serial effort. The SDK framework eliminates this by providing a shared adapter architecture: the PPRO factory alone covers 39 APMs through configuration instead of custom code. The remaining 16 use direct-integration adapters that still share the same event bus, error taxonomy, and test infrastructure.

How the Architecture Works

Every payment method in the world follows one of 6 patterns. The SDK normalizes all of them behind a single public API with two key format conversions:

Server BNPL
Klarna, Afterpay, Affirm, Sezzle, Zip, PayPal Pay Later
Widget loads -> customer approves installments -> capture
Redirect Wallet
PayPal, CashApp, Venmo, GrabPay, + PPRO wallets
Button -> redirect to wallet -> approve -> return
Bank Redirect
iDEAL, Bancontact, EPS, BLIK, Zepto, + PPRO banks
Button -> redirect to bank -> authenticate -> return
QR Code
Alipay+, WeChat Pay, Pix, UPI, TWINT
Generate QR -> customer scans with mobile -> poll for completion
Voucher / Cash
Boleto, OXXO, Efecty, Konbini, + PPRO vouchers
Generate voucher code -> customer pays offline -> webhook confirms
Native Wallet
Apple Pay, Google Pay
Browser payment sheet -> biometric auth -> tokenized payment

The flow is always: Merchant Page -> SDK -> Adapter (converts to CommerceHub format) -> CommerceHub API (converts to provider format) -> APM Provider. Responses flow back through the same two conversions. 18 universal events and 12 error codes are standardized across all 55 APMs.
